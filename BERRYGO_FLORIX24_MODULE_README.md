# BerryGo → Florix24: модуль API-интеграции

**Версия патча:** 2026-07-19  
**Назначение:** автоматическая передача новых заказов BerryGo во Florix24 и двусторонняя синхронизация основных статусов.

## Что реализовано

- новая вкладка `BerryGo Admin → Настройки → Интеграции`;
- отдельный блок подключения Florix24;
- хранение адреса Florix24, API-токена и webhook secret;
- кнопка проверки подключения;
- автоматическая постановка в очередь каждого нового заказа:
  - с публичного сайта;
  - из административной панели;
  - из Telegram-бота;
- правило `один orders.id BerryGo = один заказ Florix24`;
- `order_group_id` передаётся только как дополнительная связь;
- старые заказы не отправляются: началом синхронизации считается момент первого включения интеграции;
- двусторонняя синхронизация только четырёх статусов:
  - `new`;
  - `confirmed`;
  - `completed`;
  - `cancelled`;
- внутренние статусы BerryGo `reserved`, `shipped`, `returned` наружу не отправляются;
- входящий подписанный webhook Florix24;
- защита от повторов и бесконечного обмена статусами;
- конфликты `completed ↔ cancelled` не перезаписываются автоматически;
- фоновая очередь и повторные попытки при ошибках;
- журнал интеграции;
- отображение состояния синхронизации в карточке заказа;
- ручная кнопка повторной отправки ошибочного события.

## 1. Установка файлов

1. Сделать резервную копию файлов BerryGo и базы данных.
2. Распаковать патч поверх корня сайта BerryGo, сохранив структуру каталогов.
3. Не удалять существующие файлы конфигурации.
4. При наличии доступа к Composer выполнить из корня сайта:

```bash
composer dump-autoload -o
```

Обычно новые классы подхватываются PSR-4 и без этой команды, но обновление autoload желательно.

## 2. Обновление базы

Импортировать через phpMyAdmin или консоль файл:

```text
database/20260719_florix24_integration.sql
```

Миграция:

- не отправляет и не изменяет старые заказы;
- создаёт только служебные таблицы интеграции;
- добавляет настройки Florix24 со значением «выключено»;
- безопасна для повторного запуска благодаря `CREATE TABLE IF NOT EXISTS` и `INSERT IGNORE`.

При штатном использовании мигратора проекта возможна команда:

```bash
php bin/migrate.php up
```

Для рабочего сервера предпочтительнее импортировать конкретный SQL-файл, чтобы случайно не запустить другие необработанные миграции проекта.

## 3. Настройка BerryGo

Открыть:

```text
BerryGo Admin → Настройки → Интеграции
```

Заполнить:

```text
Адрес Florix24: https://florix24.ru
API-токен: токен из Florix24
Webhook secret: секрет из подключения Florix24
```

Включить:

```text
Отправлять новые заказы
Отправлять изменения статусов
Принимать статусы из Florix24
Повторять отправку при ошибке
```

Сначала сохранить настройки, затем нажать **«Проверить подключение»**.

У токена Florix24 должны быть права:

```text
orders.create
orders.read
orders.status.update
```

## 4. Настройка Florix24

В карточке подключения BerryGo в Florix24 указать webhook URL, который показывает BerryGo:

```text
https://berrygo.ru/api/integrations/florix24/order-status
```

В обеих системах должен быть одинаковый `Webhook secret`.

Сопоставление статусов:

| Системный статус Florix24 | Код BerryGo | Направление |
|---|---|---|
| Новый | `new` | в обе стороны |
| Принят | `confirmed` | в обе стороны |
| Выполнен | `completed` | в обе стороны |
| Отменён | `cancelled` | в обе стороны |

## 5. Планировщик BerryGo

Добавить задание каждую минуту.

Команда для текущего хостинга:

```bash
php /var/www/user674317/data/www/berrygo.ru/cron/process_florix24_queue.php
```

Экспертное расписание:

```text
Минуты:       *
Часы:         *
Дни месяца:   *
Месяцы:       *
Дни недели:   *
```

Описание:

```text
BerryGo — отправка заказов и статусов во Florix24
```

Отчёт по e-mail можно отключить.

CLI-файл использует блокировку, поэтому два экземпляра очереди одновременно не работают.

## 6. Повторные попытки

При временной ошибке запрос повторяется через:

```text
1 минуту
5 минут
15 минут
1 час
3 часа
12 часов
```

После исчерпания автоматических попыток событие остаётся в журнале со статусом `error`. Его можно повторить вручную.

Сбой Florix24 никогда не отменяет локальное создание заказа BerryGo.

## 7. Проверка после установки

### Сценарий A: заказ с сайта

1. Включить интеграцию.
2. Создать новый заказ на сайте BerryGo.
3. Убедиться, что событие появилось в журнале BerryGo.
4. Дождаться запуска cron.
5. Проверить появление заказа во Florix24.

### Сценарий B: заказ из админки

1. Создать новый заказ в BerryGo Admin.
2. Проверить, что отдельные `orders.id` ушли отдельными заказами Florix24.
3. Убедиться, что `order_group_id` не объединил их в один заказ.

### Сценарий C: статусы

1. Перевести заказ BerryGo в `confirmed` — Florix24 должен получить `confirmed`.
2. Перевести заказ во Florix24 в `completed` — BerryGo должен получить `completed`.
3. Проверить `cancelled` в обе стороны.
4. Убедиться, что `shipped` и `returned` не отправляются.

### Сценарий D: отказ Florix24

1. Временно указать неверный токен.
2. Создать заказ — он должен сохраниться в BerryGo.
3. В журнале должна появиться ошибка.
4. Вернуть правильный токен.
5. Нажать «Повторить» или дождаться очередной попытки.

## 8. Безопасность

- API-токен и webhook secret не выводятся полностью после сохранения;
- секреты не включаются в журнал событий;
- токен передаётся только сервером в заголовке `Authorization: Bearer`;
- входящие webhook проверяются HMAC-SHA256;
- допустимое расхождение времени webhook — 5 минут;
- повторный `event_id` не применяется повторно;
- HTTPS-сертификат Florix24 проверяется cURL.

## 9. Откат

1. Выключить интеграцию в настройках BerryGo.
2. Отключить задание cron.
3. Восстановить резервную копию изменённых файлов.
4. Служебные таблицы можно оставить: они не влияют на работу BerryGo при выключенной интеграции.

Не удалять таблицы интеграции до завершения диагностики: в них находится журнал отправок и ошибок.
