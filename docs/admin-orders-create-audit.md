# Аудит `/admin/orders/create`: переход на закупки и ящики

Дата аудита: 2026-05-29.

## 1. Несоответствия прежней логики

- Форма строила список из `products`, а не из `purchase_batches`, поэтому менеджер не видел реальный источник товара и остаток конкретной закупки.
- Стоимость ручного заказа считалась через `products.price * products.box_size`, то есть через цену за содержимое ящика, а не через цену партии за ящик.
- В `order_items` при ручном создании не записывались `purchase_batch_id` и `stock_mode`, поэтому дальнейшее движение товара не было связано с закупкой.
- Ручное создание не вызывало `OrderStockOrchestrator` / `StockService`, поэтому свободные ящики партии не списывались и можно было оформить больше, чем есть в закупке.
- Дата получения выбиралась до выбора закупки и не зависела от статуса партии.
- Интерфейс смешивал товар в наличии и предзаказ без явной визуальной маркировки режима.

## 2. Новый процесс оформления

1. Выбор клиента или создание клиента.
2. Выбор режима продажи:
   - **В наличии** — закупки со статусами `purchased`, `arrived`.
   - **Предзаказ** — закупки со статусом `planned`.
   - `arrived` закрепляется как статус **«Готова к выдаче / В наличии»**.
   - Если менеджер выбирает товары с разными датами получения, система должна оформить их отдельными заказами: один заказ не может содержать позиции на разные даты.
3. Выбор закупки по дате и статусу.
4. Добавление товаров выбранной закупки только в ящиках.
5. Выбор даты получения из допустимого окна:
   - в наличии: сегодня, завтра, послезавтра;
   - предзаказ: дата закупки, дата закупки +1, дата закупки +2.
6. Проверка итогов и создание заказа.

## 3. Правила расчетов и списания

- Единица продажи — **ящик**.
- Цена берется из партии:
  - `instant_price_per_box` для наличия;
  - `preorder_price_per_box` для предзаказа.
- Для товара в наличии остаток берется из `purchase_batches.boxes_free`.
- Для planned-закупок доступность предзаказа считается как плановый лимит (`boxes_total`, либо совместимый fallback `boxes_free + boxes_reserved`) минус уже созданные брони `boxes_reserved`.
- Каждая позиция заказа сохраняет `purchase_batch_id` и `stock_mode`.
- Наличие списывается сразу как продажа из свободных ящиков без промежуточной брони.
- Предзаказ создается как бронь партии, без немедленной продажи.
- Ручная форма не использует legacy-остатки `products.free_stock_boxes` для проверки доступности.

## 4. Изменяемые файлы

- `src/Views/admin/orders/create.php` — новый batch-first mobile-first интерфейс.
- `src/Services/AdminOrdersPageService.php` — подготовка списка доступных закупок для ручной формы.
- `src/Controllers/OrdersController.php` — сохранение ручного заказа через `purchase_batch_id`, цену за ящик и сервис списания остатков.
- `tests/AdminOrdersPageServiceTest.php` — проверка данных для формы создания заказа.
- `docs/admin-orders-create-audit.md` — этот аудит и план тестирования.

## 5. Схема mobile-first интерфейса

```text
[Шаг 1: Клиент]
  Телефон / подсказки / новый клиент
  Адрес или самовывоз

[Шаг 2: Режим]
  Зеленая карточка: В наличии
  Оранжевая карточка: Предзаказ

[Шаг 3: Закупка]
  Закупка 2026-05-29
  Статус: Готова к выдаче
  Всего свободно: N ящиков

[Шаг 4: Товары]
  Название SKU
  Свободно: N ящиков
  Цена: N ₽/ящик
  [-] [кол-во ящиков] [+]
  purchase_batch_id: N

[Шаг 5: Дата]
  3 кнопки допустимых дат
  Интервал получения

[Шаг 6: Проверка]
  Позиции × ящики
  Товары / скидка / баллы / доставка / итого
  Создать заказ
```

## 6. План тестирования

- Проверить, что `buildCreateData()` возвращает только партии с доступными ящиками и корректно разделяет `in_stock` / `preorder`.
- Создать заказ из закупки `arrived` и убедиться, что:
  - в `order_items` записан `purchase_batch_id`;
  - `stock_mode = instant`;
  - `boxes_free` уменьшается;
  - `boxes_sold` увеличивается;
  - `boxes_reserved` не меняется, так как товар в наличии сразу продается.
- Создать предзаказ из закупки `planned` и убедиться, что:
  - заказ получает статус `reserved`;
  - `stock_mode = preorder`;
  - партия резервируется без немедленного увеличения `boxes_sold`.
- Для planned-закупки проверить доступность как `boxes_total - boxes_reserved`, включая сценарий с `boxes_free = 0`.
- Попробовать указать больше ящиков, чем доступно по выбранному режиму, и получить ошибку `stock`.
- Проверить ограничения дат для наличия и предзаказа.
- Проверить, что позиции с разными датами получения не попадают в один заказ.
- Проверить mobile-flow на ширине 360–430 px: все основные действия доступны без горизонтального скролла.
