# Runbook: Этап 7 — QA, нагрузка и безопасность (preorder)

## 1) Функциональный QA (обязательный минимум)

### Сценарии жизненного цикла
1. `intent_created` -> `offer_sent` -> `confirmed` -> `checkout_completed`.
2. `intent_created` -> `offer_sent` -> `declined`.
3. `intent_created` -> `offer_sent` -> `expired`.

### Edge-cases
- Двойной клик по confirm/decline.
- Повторный заход по `checkout_token`.
- Истечение TTL в момент подтверждения.
- Несколько пользователей подтверждают офферы одновременно.

## 2) Нагрузочные проверки

### Что мерить
- Время выполнения волны `preorder_send_offers.php`.
- Время выполнения `preorder_expire_offers.php` на большом объеме.
- Доля ошибок напоминаний `preorder_remind_expiring.php`.

### Тест-набор
- 10k intent по одному продукту.
- 1k одновременных confirm-запросов.
- Пиковые cron-запуски remind+expire.

### Практический запуск в среде
```bash
php bin/preorder_load_probe.php <product_id> [sample_intents=1000] [available_boxes=500] [price=1200]
```
Зафиксировать `elapsed_ms`, `offered`, `allocated` и приложить к отчету QA.

## 3) Безопасность
- `checkout_token` должен быть непредсказуемым.
- Все user-facing действия должны проверять `user_id` владельца intent.
- В логах не хранить чувствительные данные (полный телефон/секреты).

### Security-проверки (подтверждение в средах)
1. Попытка подтвердить intent другого пользователя должна возвращать ошибку доступа/не найдено.
2. Токен продолжения должен иметь hex-формат и достаточную длину (48 символов для текущей реализации).
3. Повторное использование завершенного/недействительного токена должно отклоняться.

## 4) Критерии готовности этапа 7
- Нет блокирующих P1/P2 дефектов.
- Smoke-тест жизненного цикла проходит стабильно.
- Race-condition сценарии не приводят к двойной аллокации.
