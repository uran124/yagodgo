# Runbook: если упал деплой или миграция

## Когда использовать

- Деплой завершился с ошибкой.
- Ошибка на шаге применения SQL-миграций.
- После деплоя приложение отвечает 5xx или не поднимается.

## Быстрая диагностика (5–10 минут)

1. Проверить, что приложение читает корректные runtime-конфиги (`config/database.php`, `config/telegram.php`, `config/sms.php`, `config/email.php`).
2. Проверить доступ к БД и креды.
3. Посмотреть статус миграций:

```bash
php bin/migrate.php status
```

4. Сверить последнюю применённую миграцию и первый pending-файл.
5. Проверить логи приложения/веб-сервера (php-fpm/nginx/apache).

## Сценарий A: упал шаг миграции

### Симптомы

- В CI/на сервере: `Failed to apply <migration>.sql`
- В `status`: часть миграций `applied`, часть `pending`.

### Действия

1. Остановить дальнейший rollout на остальные инстансы.
2. Выполнить на целевом окружении:

```bash
php bin/migrate.php status
```

3. Открыть проблемный SQL-файл и определить тип ошибки:
   - синтаксис SQL;
   - конфликт с текущей схемой;
   - данные не соответствуют предположениям миграции.
4. Подготовить **новую** корректирующую миграцию (не переписывать уже применённые файлы).
5. Прогнать локально на копии схемы:

```bash
php bin/migrate.php up
vendor/bin/phpunit
```

6. После проверки задеплоить фикс и повторить миграции.

## Сценарий B: деплой упал до миграций

### Симптомы

- Ошибка установки зависимостей/сборки.
- Приложение не стартует после выката.

### Действия

1. Проверить логи шага `composer install` и runtime-ошибки PHP.
2. Проверить обязательные ключи окружения (DB/Telegram/SMS/Email).
3. Выполнить smoke-команды на хосте:

```bash
php -v
composer install --no-interaction --prefer-dist --optimize-autoloader
php bin/migrate.php status
```

4. Если причина не устранена быстро (SLA breach), выполнить rollback на предыдущий стабильный релиз.

## Сценарий C: после успешного деплоя появились 5xx

1. Проверить, что БД-схема совпадает с кодом (`php bin/migrate.php status` → Pending = 0).
2. Проверить права доступа к директориям логов/кэша.
3. Проверить интеграционные конфиги (Telegram/SMS/email) на предмет отсутствующих ключей.
4. Провести smoke-check основных роутов: `/`, `/login`, критичные админ/менеджер URL.

## Правила rollback

Rollback обязателен, если одновременно выполняются условия:

- инцидент влияет на прод-пользователей;
- нет подтверждённого фикс-плана < 30 минут;
- есть стабильный предыдущий релиз.

После rollback:

1. Зафиксировать таймлайн инцидента.
2. Создать postmortem с корневой причиной и preventive actions.
3. Добавить тест/чек в CI, чтобы избежать повторения.

## Коммуникация

- Обновлять статус инцидента каждые 15 минут.
- В статусе указывать: влияние, текущий шаг, ETA на следующий апдейт.
- После восстановления: короткий итог + ссылка на postmortem.
