# BerryGo Production Core и партнёрская модель

## 1. Главный принцип направления

Новый блок производства и партнёров должен **улучшать существующую модель принятия и обработки заказов**, а не создавать параллельный контур заказов.

В системе уже есть клиентские заказы, роли менеджера, партнёра и селлера, статусы, история и обработка заказов. Поэтому дальнейшая разработка должна расширять текущий order flow через производственные задания, исполнителей и модели исполнения, но не создавать отдельный независимый механизм «заказов производства» со своей жизнью, не связанной с основным заказом.

Ключевой принцип:

> Клиентский заказ остаётся единой точкой правды. Производственное задание — это операционная задача внутри заказа, а не отдельный параллельный заказ.

Это особенно важно для контроля качества и сроков: система должна гарантировать, что заказ будет выполнен вовремя и в нужном качестве, а менеджер должен видеть весь путь заказа в одном операционном контуре.

## 2. Почему не делаем отдельную биржу подрядчиков в MVP

Изначальная идея биржи подрядчиков по принципу «кто первый взял — тот выполняет» подходит для масштабирования, но слишком рискованна для первого запуска категории «Клубника в шоколаде».

Основные риски биржевой модели:

- быстрый исполнитель не всегда самый надёжный;
- качество продукта может быть нестабильным;
- сложнее контролировать сроки;
- появляется много спорных ситуаций с выплатами;
- клиент всё равно считает продавцом berryGo;
- бренд berryGo страдает при ошибке любого подрядчика;
- система распределения заказов становится сложнее самого MVP.

Для первого этапа важнее не масштаб, а управляемость:

- один понятный ответственный исполнитель;
- единый производственный стандарт;
- фото-контроль;
- прозрачная экономика заказа;
- ручной контроль менеджера;
- возможность позже расшириться до партнёров и селлеров.

## 3. Новая концепция этапа

Этап запускается как **BerryGo Production Core**.

berryGo создаёт собственную категорию «Клубника в шоколаде». Продукт продаётся под брендом berryGo, а изготовление выполняет штатный флорист или другой внутренний исполнитель.

Если исполнитель находится на смене, заказ изготавливается на рабочем месте berryGo. Если исполнитель не на смене, но доступен удалённо, berryGo может отправить ему необходимые материалы на такси, исполнитель делает заказ дома, загружает фото результата и передаёт готовый заказ курьеру или такси.

При этом архитектура должна сразу учитывать будущий рост:

- подключение внешних производственных партнёров;
- подключение селлеров со своими товарами;
- комиссии с продаж;
- абонентское размещение;
- разные модели исполнения заказа;
- разные роли продавца, бренда, исполнителя и службы доставки.

## 4. Важное разделение ролей

В системе нужно явно разделять четыре роли:

| Роль | Что означает |
| --- | --- |
| Продавец | Кто юридически/коммерчески продаёт товар клиенту |
| Владелец бренда | Под чьим брендом товар отображается клиенту |
| Исполнитель | Кто физически производит, собирает или комплектует заказ |
| Оператор доставки | Кто отвечает за передачу заказа клиенту |

Для клубники в шоколаде на первом этапе:

| Роль | Значение |
| --- | --- |
| Продавец | berryGo |
| Владелец бренда | berryGo |
| Исполнитель | штатный флорист / внутренний исполнитель |
| Оператор доставки | berryGo или назначенный курьер/такси |

Для будущего селлера:

| Роль | Возможное значение |
| --- | --- |
| Продавец | селлер или berryGo, зависит от договора |
| Владелец бренда | селлер |
| Исполнитель | селлер |
| Оператор доставки | berryGo или селлер |

## 5. Типы партнёров

Общий раздел лучше называть не только «Селлеры», а **«Партнёры и производство»**. Внутри можно выделять типы участников.

Рекомендуемые типы партнёров:

| Тип | Описание | Видим клиенту |
| --- | --- | --- |
| `internal_staff` | штатный сотрудник berryGo | нет |
| `production_partner` | внешний производитель под брендом berryGo | нет |
| `marketplace_seller` | самостоятельный селлер со своими товарами | да, если выбрана публичная модель |
| `brand_partner` | бренд или магазин-партнёр | зависит от договора |

Для текущего этапа нужен только `internal_staff`, но модели должны не мешать добавлению остальных типов позже.

## 6. Модели исполнения заказа

Рекомендуемые модели исполнения:

| Модель | Описание |
| --- | --- |
| `by_berrygo_on_site` | делает сотрудник berryGo на рабочем месте |
| `by_berrygo_remote` | делает сотрудник berryGo удалённо, материалы доставляются ему отдельно |
| `by_partner_under_berrygo_brand` | внешний партнёр делает товар под брендом berryGo |
| `by_seller` | селлер сам производит/собирает и передаёт заказ |
| `by_berrygo_from_seller_stock` | товар селлера хранится/комплектуется у berryGo |

Для клубники в шоколаде на первом этапе используются:

- `by_berrygo_on_site`;
- `by_berrygo_remote`.

## 7. Модели монетизации

Рекомендуемые модели монетизации:

| Модель | Описание |
| --- | --- |
| `salary` | работа входит в зарплату сотрудника |
| `internal_bonus` | внутренний бонус сотруднику за выполненный заказ |
| `fixed_payout` | фиксированная выплата исполнителю за заказ |
| `commission` | комиссия berryGo с продажи селлера |
| `subscription` | абонентская плата за размещение |
| `commission_plus_subscription` | комиссия плюс абонентка |
| `fixed_fee_per_order` | фиксированная плата за каждый заказ |

Для клубники в шоколаде используется `internal_bonus`.

Базовое правило бонуса:

```text
bonus = 10% от продажной цены товара
```

Более безопасный вариант для экономики и мотивации:

```text
bonus = max(фиксированная сумма за товар, 10% от продажной цены товара)
```

## 8. MVP: клубника в шоколаде как внутреннее производство

### Цель MVP

Запустить категорию «Клубника в шоколаде» как собственный продукт berryGo без отдельного цеха и без внешней биржи подрядчиков.

### В MVP входит

1. SEO-категория `/klubnika-v-shokolade`.
2. Небольшая линейка стандартных товаров berryGo.
3. Производственные стандарты для каждого товара.
4. Производственные задания внутри существующего заказа.
5. Назначение задания внутреннему исполнителю.
6. Учет режима исполнения: на смене или удалённо.
7. Учет материалов для изготовления.
8. Учет доставки материалов исполнителю при удалённом производстве.
9. Учет доставки готового результата от исполнителя.
10. Фото-контроль готового товара.
11. Ручное подтверждение менеджером для первых/дорогих/проблемных заказов.
12. Расчёт бонуса исполнителю.
13. Расчёт маржи заказа с учётом материалов, такси, бонуса и эквайринга.
14. История ключевых производственных событий внутри заказа.

### В MVP не входит

- открытая биржа подрядчиков;
- конкуренция исполнителей за заказ;
- самостоятельная регистрация селлеров;
- публичные профили селлеров;
- сложный рейтинг партнёров;
- автоматические штрафы;
- торги за заказ;
- мобильное приложение для исполнителей;
- полноценная складская система материалов.

## 9. Производственное задание

Производственное задание — это внутренняя операционная задача, связанная с клиентским заказом.

Оно не заменяет заказ и не должно жить отдельно от него.

Карточка производственного задания должна содержать:

- номер клиентского заказа;
- товар;
- количество;
- эталонное фото;
- производственный стандарт;
- дедлайн начала производства;
- дедлайн готовности;
- дедлайн передачи курьеру;
- назначенного исполнителя;
- модель исполнения;
- место производства;
- список материалов;
- необходимость доставки материалов;
- стоимость доставки материалов;
- необходимость доставки готового результата;
- стоимость доставки готового результата;
- сумму бонуса исполнителю;
- текущий статус;
- фото результата;
- комментарии менеджера;
- историю событий.

## 10. Статусы производственного задания

Базовые статусы:

```text
new
assigned
materials_pending
materials_sent
materials_received
in_progress
photo_uploaded
approved
ready_for_handover
handed_over
completed
cancelled
problem
```

Для производства на смене цепочка может быть короткой:

```text
new → assigned → in_progress → photo_uploaded → approved → handed_over → completed
```

Для удалённого производства цепочка длиннее:

```text
new → assigned → materials_pending → materials_sent → materials_received → in_progress → photo_uploaded → approved → handed_over → completed
```

## 11. Производственные стандарты

Для каждого товара в категории «Клубника в шоколаде» нужен производственный стандарт.

Стандарт должен содержать:

- эталонное фото;
- состав;
- количество ягод;
- допустимый размер ягод;
- вид шоколада;
- декор;
- упаковку;
- цвет ленты;
- правила открытки;
- срок изготовления;
- условия хранения;
- инструкцию по фото;
- инструкцию по передаче курьеру;
- список допустимых замен;
- список запрещённых замен.

Это не маркетинговое описание товара, а техническое задание для исполнителя и менеджера.

## 12. Фото-контроль

Перед передачей заказа исполнитель обязан загрузить фото результата.

Минимальный набор фото:

- общий вид набора сверху;
- крупный план ягод;
- фото набора в упаковке;
- фото открытки, если она есть;
- фото перед передачей курьеру, если требуется.

Менеджер может:

- принять фото;
- отклонить фото;
- попросить переделать;
- поставить заказ в статус `problem`;
- разрешить передачу курьеру.

Для первых заказов нового исполнителя, дорогих заказов и заказов с жалобами ручная проверка обязательна.

## 13. Удалённое производство

Удалённое производство разрешается только если:

- исполнитель доступен;
- есть достаточный запас времени;
- можно доставить материалы;
- можно забрать или доставить готовый результат;
- прогнозная маржа заказа выше минимального порога;
- менеджер видит и контролирует процесс.

Нужно учитывать расходы:

- такси с материалами до исполнителя;
- такси или курьер от исполнителя до точки передачи/клиента;
- возможные потери материалов;
- дополнительное время на логистику.

Правило безопасности:

> Если удалённое производство делает заказ убыточным или слишком рискованным по времени, система должна предложить другой слот, выполнение на смене или ручное решение менеджера.

## 14. Экономика заказа

Для заказа клубники в шоколаде нужно видеть расчёт:

```text
цена клиента
скидка / промокод
стоимость материалов
стоимость доставки материалов
стоимость доставки готового результата
бонус исполнителю
эквайринг
прочие расходы
итоговая маржа berryGo
```

Минимальный порог маржи должен быть настраиваемым.

Если прогнозная маржа ниже порога, заказ должен подсвечиваться менеджеру или не должен автоматически уходить в удалённое производство.

## 15. Защита от параллельного принятия и параллельной обработки

Даже если в MVP нет конкуренции исполнителей, система должна быть готова к будущему появлению нескольких исполнителей и селлеров.

Поэтому нельзя строить логику по принципу «сначала прочитали статус, потом записали нового исполнителя».

Назначение или принятие производственного задания должно быть атомарным действием:

```sql
UPDATE production_jobs
SET
    executor_id = :executor_id,
    status = 'assigned',
    assigned_at = NOW()
WHERE id = :job_id
  AND executor_id IS NULL
  AND status = 'new';
```

Успех определяется по числу изменённых строк:

- изменилась одна строка — задание назначено;
- не изменилась ни одна строка — задание уже недоступно или изменило статус.

Этот принцип должен применяться ко всем будущим сценариям:

- назначение исполнителя;
- принятие задания селлером;
- резервирование задания;
- возврат задания в работу;
- переходы критических статусов.

## 16. Встраивание в существующий order flow

Новая разработка должна расширять текущую обработку заказов.

Требования:

1. Клиентский заказ остаётся главным объектом.
2. Производственное задание создаётся как дочерняя сущность заказа.
3. Статусы производства не должны конфликтовать с основными статусами заказа.
4. Менеджер должен видеть производство из карточки заказа.
5. История производственных событий должна попадать в общую историю заказа или быть доступной из неё.
6. Отмена производственного задания не должна автоматически отменять клиентский заказ.
7. При проблеме производства заказ должен переходить в управляемое состояние, где менеджер принимает решение.
8. В будущем seller-заказы также должны подключаться к этому контуру, а не создавать отдельный независимый процесс.

## 17. Будущий раздел «Партнёры и селлеры»

После MVP можно развивать отдельный раздел управления партнёрами и селлерами.

Раздел должен поддерживать:

- список партнёров;
- тип партнёра;
- модель исполнения;
- модель монетизации;
- комиссию;
- абонентскую плату;
- товары партнёра;
- статусы модерации;
- качество и ограничения;
- выплаты;
- правила доставки;
- видимость селлера клиенту.

Рекомендуемые статусы селлера:

```text
draft
onboarding
moderation
active
paused
limited
blocked
archived
```

## 18. Будущие коммерческие модели селлеров

Для внешних селлеров можно поддержать несколько моделей.

### Комиссия с продажи

berryGo получает процент с оплаченного заказа.

Пример:

```text
commission_percent = 30
```

### Абонентское размещение

Селлер платит фиксированную сумму за размещение товаров.

### Комиссия плюс абонентка

Смешанная модель для приоритетных селлеров или расширенного продвижения.

### Фиксированная плата за заказ

berryGo удерживает фиксированную сумму за каждый заказ.

### Товар под брендом berryGo

Селлер или партнёр производит товар по стандарту berryGo и получает фиксированную выплату, но клиент видит berryGo.

## 19. Этапы разработки

### Этап 1. Production Core

Цель: добавить производственные задания как расширение существующего заказа.

Результат:

- есть базовая сущность производственного задания;
- задание связано с заказом;
- менеджер видит задание из заказа;
- есть базовые статусы производства;
- есть история событий.

### Этап 2. Клубника в шоколаде

Цель: запустить категорию как собственный продукт berryGo.

Результат:

- есть SEO-категория;
- есть 4–6 стартовых товаров;
- есть производственные стандарты;
- заказы создают производственные задания;
- задания назначаются внутреннему исполнителю.

### Этап 3. Внутренний исполнитель и бонусы

Цель: поддержать штатного флориста как исполнителя.

Результат:

- исполнитель может быть назначен на задание;
- учитывается модель на смене / удалённо;
- рассчитывается бонус;
- выплаты видны менеджеру.

### Этап 4. Материалы и удалённое производство

Цель: безопасно поддержать изготовление вне смены.

Результат:

- фиксируются нужные материалы;
- фиксируется отправка материалов;
- учитываются расходы на такси;
- считается маржа;
- удалённое производство блокируется при плохой экономике или риске по срокам.

### Этап 5. Фото-контроль и качество

Цель: гарантировать визуальный стандарт товара.

Результат:

- исполнитель загружает фото;
- менеджер подтверждает или отклоняет фото;
- фото сохраняются в заказе;
- проблемные заказы подсвечиваются.

### Этап 6. Партнёрская основа

Цель: подготовить систему к селлерам и партнёрам.

Результат:

- есть типы партнёров;
- есть модели исполнения;
- есть модели монетизации;
- текущий внутренний исполнитель описывается через те же универсальные принципы, что будущие партнёры.

### Этап 7. Seller Platform

Цель: дать внешним селлерам возможность продавать через berryGo.

Результат:

- появляется раздел селлеров;
- селлеры могут иметь товары;
- заказы селлеров проходят через общий order flow;
- berryGo может брать комиссию, абонентку или смешанную оплату;
- селлеры не ломают внутреннюю производственную модель.

## 20. Критерии готовности MVP

MVP направления готов, когда:

- клиент может заказать клубнику в шоколаде;
- заказ остаётся в едином существующем order flow;
- по заказу создаётся производственное задание;
- задание назначается внутреннему исполнителю;
- система понимает, делается заказ на смене или удалённо;
- при удалённом производстве фиксируются материалы и расходы доставки;
- исполнитель загружает фото результата;
- менеджер может принять или отклонить результат;
- бонус исполнителю рассчитывается корректно;
- менеджер видит маржу заказа;
- отмена или проблема производства не уничтожает клиентский заказ;
- критические действия фиксируются в истории;
- архитектура готова к будущим селлерам и партнёрам.

## 21. Итоговая формулировка roadmap

**Этап: BerryGo Production Core — внутреннее производство и основа партнёрской платформы.**

Результат этапа:

> berryGo запускает клубнику в шоколаде как собственный продукт, производимый внутренним исполнителем по стандарту berryGo, с контролем сроков, качества, фото результата, материалов, удалённого изготовления, бонусов и маржи. Новый производственный контур улучшает существующую обработку заказов и готовит систему к будущим селлерам, не создавая параллельную модель заказов.
