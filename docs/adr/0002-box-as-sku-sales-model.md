# ADR-0002: Единая модель продажи «товар = ящик (SKU)»

- **Дата:** 2026-05-27
- **Статус:** Accepted

## Контекст

Сейчас в проекте одновременно используются две несовместимые бизнес-модели:

1. Старая: товаром считается содержимое ящика.
2. Новая: товаром считается сам ящик, а вес/объём — характеристика SKU.

Это приводит к инцидентам вида «товар отображается доступным на главной, но не добавляется в корзину», потому что разные части системы читают цену/остаток из разных мест.

## Решение

### 1) Каноническая модель продажи

1. **Единица продажи = ящик.**
2. **Количество в корзине/заказе = количество ящиков.**
3. **Вес/объём ящика — характеристика SKU, а не отдельная единица продажи.**

Пример:

- Товар: `Клубника Азия, ящик 2 кг`
- Цена: `1500 ₽ за ящик`
- В корзине: `2 ящика`
- Итого: `3000 ₽`

### 2) Определение SKU

Каждый тип ящика (вес/объём) — отдельный SKU и отдельная карточка товара.

- `Клубника Азия 2 кг` = SKU A
- `Клубника Азия 3 кг` = SKU B

Запрещается возвращаться к схеме «один product + размеры через опции», если это ломает инвариант «количество считается в ящиках SKU».

### 3) Разделение ролей таблиц

#### `products` хранит только карточку SKU

- название
- описание
- характеристики ящика
- медиа
- SEO
- настройки отображения

**Таблица `products` не используется как источник актуального остатка, доступности или цены.**

#### `purchase_batches` — источник операционной коммерческой логики

- сколько ящиков пришло (`boxes_total`)
- сколько ящиков свободно (`boxes_free`)
- цена за ящик (`price_per_box`)
- цена уценки (`discount_price_per_box`)
- статус партии (`active` / `discount` / `preorder` / ...)
- служебные поля выбора партии (дата/приоритет)

### 4) Единый источник истины

**Наличие, цена и списание определяются только через `purchase_batches`.**

Запрещается двусторонняя синхронизация остатков между `products` и `purchase_batches`.

Legacy-поля `products` (например, `free_stock_boxes`, `discount_stock_boxes`, `instant_price_per_box`, `preorder_price_per_box`, `current_purchase_batch_id`) могут существовать только как временная проекция для совместимости и не участвуют в принятии бизнес-решений.

### 5) Стратегия выбора партии (единая для всех read/write-path)

До изменения кода нужно зафиксировать одну стратегию в конфигурации доменного сервиса и использовать её везде (каталог, карточка, корзина, checkout):

- либо **FIFO**: продаём самую старую подходящую активную партию;
- либо **priority-first**: берём партию с максимальным приоритетом (при равенстве — детерминированный tie-break по дате/ID).

### 6) Политика цены в корзине

Корзина хранит snapshot цены:

- `purchase_batch_id`
- `boxes_qty`
- `price_per_box_snapshot`
- `price_type` (`regular` / `discount` / `preorder`)

Это исключает неожиданное изменение цены в уже собранной корзине.

### 7) Atomic списание остатков

Все операции резервирования/списания остатков выполняются только внутри транзакции БД с блокировкой строк партии (`SELECT ... FOR UPDATE` или эквивалент).

Цель: не допустить oversell при конкурентном checkout.

### 8) Временная совместимость (compatibility layer)

На переходный период допускается однонаправленное обновление legacy-полей для:

- старой админки,
- старых отчётов,
- старых API.

Но бизнес-логика (доступность, цена, добавление в корзину, checkout) не читает legacy-поля.

## Инварианты для code review

1. Нельзя проверять остатки/цены через поля `products`.
2. Любой add-to-cart обязан привязываться к `purchase_batch_id`.
3. Любой checkout обязан списывать ящики из `purchase_batches` в транзакции.
4. Количества в доменной логике выражаются только в ящиках.
5. Каталог/карточка/корзина должны использовать один и тот же batch-resolver.

## Definition of Done

Задача миграции считается выполненной, если:

1. Товар в наличии всегда добавляется в корзину.
2. Главная, карточка и корзина используют один источник наличия/цены (`purchase_batches`).
3. Checkout списывает остатки только из `purchase_batches`.
4. Невозможно продать больше ящиков, чем есть в остатке.
5. Все операции работают в ящиках.
6. Legacy-поля не участвуют в принятии решений.
7. Пройдены e2e сценарии продаж, включая конкурентные кейсы.
8. Не наблюдается рассинхронизация между каталогом и корзиной после релиза.

## План следующих этапов (после ADR)

1. Аудит всех read/write по legacy-полям и карта мест использования.
2. Внедрение единого batch-resolver и переключение каталога/карточки.
3. Переписывание add-to-cart на batch-first + snapshot цены.
4. Перевод checkout на atomic списание по batch.
5. Включение compatibility layer и поэтапная депрекация legacy-полей.
6. E2E и нагрузочные проверки anti-oversell, затем релиз под feature flag.
