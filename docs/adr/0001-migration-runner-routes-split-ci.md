# ADR-0001: Migration runner + split routes + CI

- **Дата:** 2026-03-30
- **Статус:** Accepted

## Контекст

Проект рос итеративно, и три зоны начали тормозить команду:

1. Изменения схемы БД применялись вручную, что приводило к расхождениям окружений.
2. Роуты в одном месте ухудшали читаемость и усложняли ревью.
3. Перед merge не было единообразной автоматической проверки миграций и тестов.

## Решение

1. **Ввести migration runner** (`bin/migrate.php`) с командами:
   - `up`
   - `status`
   - `up --dry-run`
2. **Разделить роутинг по зонам ответственности** в каталоге `routes/` (`public.php`, `client.php`, `admin.php`, `seller.php`, и т.д.), оставив `index.php` точкой входа и оркестратором подключения.
3. **Зафиксировать обязательный CI-пайплайн** (`.github/workflows/ci.yml`): install dependencies → поднятие MySQL → прогон миграций → phpunit.

## Почему так

- Migration runner делает изменение схемы **повторяемым** и уменьшает риск “у меня работает / на сервере нет”.
- Split routes уменьшает когнитивную нагрузку и даёт локальность правок по ролям/контекстам.
- CI даёт быстрый сигнал о поломках до прода и делает качество merge воспроизводимым.

## Последствия

### Плюсы

- Онбординг проще: одна последовательность шагов для локального старта.
- Деплой безопаснее: есть статус миграций и dry-run.
- Ревью быстрее: меньше монолитных роут-файлов.
- Merge-дисциплина выше: обязательные проверки в CI.

### Минусы / trade-offs

- Нужно поддерживать дисциплину именования SQL-файлов и порядок миграций.
- Появляется дополнительная документация и runbook, которые нужно актуализировать.
- CI увеличивает время обратной связи (в обмен на стабильность).

## Что отклонили

- Полноценный внешне-зависимый migration framework (лишняя сложность для текущего объёма проекта).
- Полный отказ от split routes в пользу одного файла (хуже масштабируемость).
- Ручные pre-merge проверки без CI (слишком высокий риск человеческой ошибки).
