# Жизненный цикл статусов заказа

Первый этап внедрения фиксирует единую карту статусов заказа для всех разделов сайта:

| Code | Название | Смысл |
| --- | --- | --- |
| `reserved` | Бронь | Предзаказ оформлен и учитывается как потребность/дефицит. |
| `new` | Ожидает подтверждения | Клиент оформил заказ, оплата закрыта, менеджер ещё не подтвердил. |
| `confirmed` | Подтверждён | Заказ подтверждён, оплата открыта. |
| `shipped` | В пути | Заказ передан курьеру. |
| `completed` | Выполнен | Заказ успешно передан покупателю или выдан при самовывозе. |
| `cancelled` | Отменён | Заказ не состоялся до успешного выполнения. |
| `returned` | Возврат | Полный возврат заказа; товар уходит в списание, бонусы и выплаты откатываются в следующих этапах внедрения. |

Миграция старых статусов:

| Старый code | Новый code |
| --- | --- |
| `processing` | `confirmed` |
| `assigned` | `shipped` |
| `delivered` | `completed` |

Оплата не является статусом заказа и должна развиваться отдельной моделью платежных статусов и способов оплаты.

## История статусов

Каждое изменение статуса через админский/селлерский интерфейс должно записываться в `order_status_history` с исходным статусом, новым статусом, пользователем, ролью и временем изменения.

## Оплата отдельно от статуса заказа

Статус заказа не используется как признак оплаты. Для платежей используются отдельные значения:

| Code | Название |
| --- | --- |
| `unpaid` | Не оплачен |
| `pending` | Ожидает оплаты |
| `paid` | Оплачен |
| `failed` | Ошибка оплаты |
| `refund_pending` | Ожидает возврата |
| `refunded` | Возвращён |
| `partially_refunded` | Частично возвращён |

Способ оплаты хранится отдельно в `payment_method`: `online_robokassa`, `cash_on_delivery`, `cash_pickup`, `card_on_delivery`, `card_pickup`.

## Складские движения этапа `new → confirmed`

Обычный заказ в статусе `new` только сохраняет позиции заказа и не меняет остатки. При первом переходе из `new` в `confirmed`, `shipped` или `completed` позиции фиксируются через резерв товара. Если по обычному товару в наличии свободных ящиков не хватает, резерв может увести `boxes_free` в минус — это сигнал будущему блоку дефицита. При переводе в `completed` зарезервированные ящики переводятся в продажу.
