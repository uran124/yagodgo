# Полная дорожная карта внедрения предзаказа

Документ описывает все этапы перехода к модели:
`intent -> offer -> confirm/decline/expire -> checkout -> post-order analytics`.

---

## Этап 1 — бизнес-правила и терминология (согласовано)

### 1.1 Цель
Синхронизировать продукт, закупку и разработку по единой логике предзаказа.

### 1.2 Ключевые договоренности
- Лимитов на клиента нет.
- Подтверждение оффера: 4 часа.
- До поступления партии кнопка «Предзаказ -10%» создает намерение (`intent_created`), а не финальный заказ.

### 1.3 Состояния
1. `intent_created`
2. `offer_sent`
3. `confirmed`
4. `declined`
5. `expired`
6. `checkout_completed`

### 1.4 Распределение объема
- FIFO по времени создания намерения.
- При `declined/expired` объем переходит следующему в очереди.

### 1.5 Коммуникация
> Здравствуйте! Получили свежую партию **[Товар]**.
> У вас предварительный заказ: **[Количество]**, цена: **[Цена за коробку]**.
> Подтверждаете? Ответьте в течение **4 часов**.

### 1.6 DoD этапа 1
- Согласованы статусы и переходы.
- Согласованы FIFO и TTL=4ч.
- Согласован текст уведомлений и дисклеймеры.

## Этап 2 — архитектура и спецификация реализации (согласовано)

### 2.1 Изменения БД
#### Таблица `preorder_intents`
- `id` BIGINT PK
- `user_id` BIGINT NOT NULL
- `product_id` BIGINT NOT NULL
- `requested_boxes` DECIMAL(10,2) NOT NULL
- `status` ENUM('intent_created','offer_sent','confirmed','declined','expired','checkout_completed') NOT NULL
- `offered_price_per_box` DECIMAL(10,2) NULL
- `offer_expires_at` DATETIME NULL
- `checkout_token` VARCHAR(64) NULL
- `created_at` DATETIME NOT NULL
- `updated_at` DATETIME NOT NULL

Индексы:
- `(product_id, status, created_at)`
- `(user_id, product_id, status)`
- `(offer_expires_at, status)`
- `UNIQUE(checkout_token)`

Инварианты:
- `requested_boxes > 0`
- Для `offer_sent` обязательны `offered_price_per_box` и `offer_expires_at`

### 2.2 API
- `POST /preorder-intents` — создать/обновить намерение
- `POST /preorder-intents/{id}/confirm` — подтвердить оффер
- `POST /preorder-intents/{id}/decline` — отказаться
- `GET /preorder/continue/{checkout_token}` — продолжить оформление

### 2.3 Jobs
- `expire_preorder_offers`
- `reallocate_expired_or_declined`
- `remind_offer_expiring`

### 2.4 DoD этапа 2
- Миграции + индексы готовы.
- Endpoint’ы и jobs реализованы.
- Базовые интеграционные тест-кейсы описаны.

## Этап 3 — Backend implementation (код)

### 3.1 Миграции
- Создать миграцию `preorder_intents`.
- Добавить индексы и проверки консистентности.
- Добавить idempotency-ключ для confirm/decline (если будет отдельная таблица событий).

### 3.2 Доменная логика
- `PreorderIntentService`:
  - createOrUpdateIntent(userId, productId, requestedBoxes)
  - sendOffersForBatch(productId, batchId, availableBoxes, offeredPrice)
  - confirmIntent(intentId, userId)
  - declineIntent(intentId, userId)
  - expireOffers(now)
  - reallocate(productId)

### 3.3 Интеграция с закупкой
- Хук/событие после фиксации партии закупщиком:
  - расчёт доступного объёма на волну;
  - отправка офферов FIFO;
  - запись дедлайнов подтверждения.

### 3.4 Надежность
- Транзакции и блокировки на распределение объёма.
- Защита от гонок подтверждений.
- Идемпотентные confirm/decline.

### 3.5 DoD этапа 3
- Unit-тесты сервиса предзаказа.
- Интеграционные тесты сценариев распределения и истечения TTL.
- Логи переходов статусов во всех сценариях.

## Этап 4 — Frontend implementation (код)

### 4.1 Карточка товара
- Кнопка «Купить сейчас» (текущий поток).
- Кнопка «Предзаказ -10%»:
  - при отсутствии поставки: отправка `POST /preorder-intents`, success-state;
  - при наличии оффера: переход на экран подтверждения.

### 4.2 Экран подтверждения оффера
- Показать товар, количество, цену и дедлайн (4 часа).
- Кнопки «Да, подтверждаю» / «Нет».
- Обработка состояний `expired` и повторного открытия ссылки.

### 4.3 Продолжение checkout
- По `checkout_token` предзаполнить корзину.
- Дать выбрать количество, дату, способ получения.
- Довести до стандартного оформления заказа.

### 4.4 DoD этапа 4
- UX-цепочка проходит без ручных действий разработчика.
- Ошибки/edge cases корректно отображаются.
- Тексты уведомлений и экранов согласованы с продуктом.

## Этап 6 — Категория «Выгодные остатки»

### 6.1 Каталог/витрина
- Отдельная категория/блок на главной.
- Товары попадают туда после админ-действия `move-to-discount`.

### 6.2 Ограничения
- Для `discount_stock` бонусы и купоны отключены.
- Явная маркировка, что это отдельный режим покупки.

### 6.3 DoD этапа 6
- Категория отображается и фильтруется отдельно.
- Ограничения бонусов/купонов применяются стабильно.

## Этап 8 — Запуск (go-live) и постепенное масштабирование

### 8.1 Стратегия релиза
- `Phase A`: сбор intent без офферов (тихий прогрев).
- `Phase B`: ручной запуск офферов из админки.
- `Phase C`: авто-волны + авто-переаллокация.
- `Phase D`: оптимизация конверсии (напоминания, A/B тексты).

### 8.2 Метрики
- Конверсия intent -> offer
- Конверсия offer -> confirmed
- Конверсия confirmed -> checkout_completed
- Доля `expired` и `declined`
- Доля перераспределённого объёма

### 8.3 DoD этапа 8
- Метрики собираются в дашборд.
- Есть weekly review и backlog улучшений.
